## 9. 【重要】現行は既にGoogleフォーム運用（F4/F7の前提が変わる）

検索の結果、**このLOMは出欠もアンケートも既にGoogleフォームで月次運用**していることが判明（最新は2026年8月分まで）。命名規約は `26MM_…`（26=2026年, MM=月）。

- `26MM_出欠回答フォーム`（例会の出欠）
- `26MM_対内アンケート` / `26MM_対外アンケート`（例会後アンケート）
- `26MM_エントリーフォーム`（上程エントリー）/ `お申し込みフォーム` / `オブザーブお申込フォーム`
- 集計結果は `アンケート結果.pdf` / スプレッドシートで保存

→ これは**現行ワークフローそのもの**。エージェントは「ゼロから作る」のではなく、**既存フォーム資産と接続/置換**する設計が現実的（後述の設計判断）。Forms APIで設問・回答とも機械取得可能なことを実証済み（forms.googleapis.com 有効化済み）。

### 9.1 出欠フォームの実構造（F4）
最新 `2608_出欠回答フォーム` をForms APIで取得：
- 設問: 氏名 / ふりがな / 所属等（五役・各委員会）/ 役職名（理事長・監事・副理事長・専務理事・委員長・副委員長・委員）/ **出欠（出席・欠席）**
- 条件分岐セクション: 出席→「遅刻早退の予定（全日程参加 等）」、欠席→「欠席理由（体調不良/家庭の事情/仕事/冠婚葬祭）」
- 自由記述: 「学びたいこと」
- 旧 `委員会出欠表` では出欠3値（**○出席 / △Web出席 / ✕欠席**）＋個人別 出席回数・出席率を集計。

→ **F4反映**: 出欠ステータスは実運用に合わせ **出席 / Web出席 / 欠席**（+ 遅刻早退、欠席理由、委任、未回答）。集計は「各回の出席人数・個人別出席率」を標準出力に。

### 9.2 アンケートの実構造（F7）
`2604_対内アンケート` をForms APIで取得：
- 識別: 氏名 / 所属等 / 役職名
- テーマ別セクション（目的達成度 / 事業運営 / メンバー意識）
- 設問形式: **5段階スケール（Likert 1-5）** ＋ **自由記述（PARA）** の混在
- 目的明記: 「**事業目的達成の検証**にあたってのアンケート」→ 事業計画書「09 目的達成の検証」と直結
- **対内（会員向け）/ 対外（来場者・非会員向け）** の2系統が存在

→ **F7反映**: 設問テンプレ＝「識別＋Likert＋自由記述」。集計はLikertの定量平均＋自由記述のGemini要約/分類。**対外アンケートは非会員が対象＝LINE不可**のためフォーム必須。

## 12. 設計上の重要判断（要ユーザー合意）

現行のGoogleフォーム運用が成熟しているため、F4(出欠)/F7(アンケート)の実装方式に選択肢がある：

- **案A 既存フォーム連携**: エージェントが従来形式のフォームを用意/リンク配布し、Forms APIで回答を読み、名簿と突合して**未回答者をLINEで催促**。現行UX維持・低リスク。
- **案B LINEネイティブ**: 出欠・アンケートをLINEのクイックリプライで完結、DBを正に。LINE体験は最良だが既存運用を置換。
- **案C ハイブリッド（推奨）**: 高頻度・単純な**出欠はLINEネイティブ**（即時回答＋自動催促が活きる）、**アンケート（特に対外＝非会員）はフォーム維持**＋エージェントが回答取得・集計・催促。対外対象はLINE不可のためフォームが必須。
