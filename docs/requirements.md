# 青年会議所 専務理事AIエージェント 要件定義書（ドラフト v0.1）

最終更新: 2026-06-20

---

## 1. 背景・目的

青年会議所（JC）の**専務理事**は、会の事務局運営の中核として「締切・抜け漏れ・連絡の交通整理」を担う役割である。本プロジェクトは、その定型的・反復的な事務作業をAIエージェントに委譲し、メンバーとは**LINE公式アカウント（Messaging API）**を通じてやり取りすることで、専務理事および役員の負担軽減と、会運営の品質・スピード向上を図る。

### 目的（達成したい状態）
- 出欠確認・催促の自動化により、回答率と把握精度を上げ、手作業の集計をなくす。
- ブロック協議会等からの対外連絡を、漏れなく適切な対象へ確実に伝達する。
- 各種会議体（理事会・五役会・委員会）の資料提出状況を可視化し、未提出を自動催促し、内容の不備を事前に検出する。
- 例会アンケートの集計・未提出催促を自動化し、PDCAを回しやすくする。

---

## 2. スコープ

### 2.1 対象範囲（確定）
- **単一LOM（単会）**、メンバー規模〜数十名。
- ブロック協議会・地区協議会・全国は「**対外連絡の受信元**」として扱う（本エージェントが直接管理する組織ではない）。
- データモデルは将来の複数LOM化を阻害しない設計とする（`lom_id` を内部的に保持）が、機能としてのマルチテナントは当面スコープ外。

### 2.2 In Scope（本フェーズで作るもの）
- LINE経由のメンバー対話・通知配信基盤
- メンバー名簿・組織（委員会・役職）の管理
- イベント／会議スケジュールの管理
- 出欠確認・集計・自動催促
- 対外連絡（ブロック協議会等）の受領・整形・伝達
- 会議資料の提出管理 ＋ Geminiによる内容レビュー
- 例会アンケートの実施・集計・未提出催促
- 管理者向けダッシュボード（最小限）
- 全自動配信エンジン＋安全ガードレール

### 2.3 Out of Scope（当面やらないこと）
- 会計・経理処理、会費徴収・決済
- 正式な議事録の自動作成（将来拡張候補）
- 複数LOMのマルチテナント運用
- LINE以外のチャネル（メール／Webアプリ）でのメンバー向けUI（管理者向け管理画面は除く）
- 全国組織の公式システムとの双方向API連携（当面はエクスポート取込のみ想定）

## 3. 利用者・ロール

| ロール | 説明 | 主な接点 |
|---|---|---|
| 一般メンバー | 出欠回答・アンケート回答・問い合わせ | LINE |
| 委員長／委員会担当 | 資料提出、自委員会の状況確認 | LINE／管理画面 |
| 五役（理事長・専務・副理事長等） | 全体状況の把握、設定変更、例外承認 | 管理画面／LINE |
| 専務理事（人間の最終責任者） | 運用監督、ガードレール設定、キルスイッチ | 管理画面 |
| システム管理者 | 名簿・連携・インフラ運用 | 管理画面／GCPコンソール |

> ※ 全自動配信のため日常運用に人間の承認は不要だが、**監督者（専務理事）が随時介入・停止できる**ことを前提とする。

## 4. 前提・制約

- **チャネル**: LINE公式アカウント（Messaging API）。Push通知 ＋ Webhook受信 ＋ リッチメニュー ／ クイックリプライ／（必要に応じ）LINEログイン。
- **インフラ**: GCP。
- **LLM**: GCP上のGemini（Vertex AI / Google AI Studio系）。
- **正データ**: GCP上のDB（Firestore想定）を「正」とし、Google Workspace（Sheets/Calendar/Drive/Gmail）を入出力連携先とするハイブリッド。
- **個人情報**: メンバーの氏名・連絡先・LINE IDを扱うため、個人情報保護法および会の規程に準拠。
- **コスト**: 数十名規模・サーバレス中心で月額を低く抑える（後述コスト想定）。

## 5. 全体アーキテクチャ（GCP想定）

```
[LINEメンバー]
     │  (LINE Messaging API / Webhook)
     ▼
[Cloud Run: LINE Webhook受信 & 対話API]──┐
     │                                    │
     ▼                                    ▼
[Cloud Run: コアロジック/ジョブAPI]   [Vertex AI: Gemini]
     │        │            │
     ▼        ▼            ▼
[Firestore] [Cloud Tasks] [Cloud Storage(資料)]
     ▲           ▲
     │           │ (定期トリガ)
[Google Workspace連携]   [Cloud Scheduler]
 (Sheets/Calendar/Drive/Gmail)
     ▲
     │
[管理ダッシュボード(Cloud Run + IAP保護)]
```

| 役割 | GCPサービス（想定） |
|---|---|
| アプリ実行（Webhook/API/UI） | Cloud Run |
| DB（メンバー・イベント・回答等） | Firestore（または Cloud SQL） |
| 非同期ジョブ・配信キュー | Cloud Tasks |
| 定期実行（催促・締切判定） | Cloud Scheduler |
| ファイル保管（会議資料・添付） | Cloud Storage |
| LLM推論 | Vertex AI（Gemini） |
| シークレット管理 | Secret Manager |
| 認証・管理画面保護 | Identity-Aware Proxy（IAP）／Firebase Auth |
| ログ・監視・通知 | Cloud Logging / Monitoring / Error Reporting |
| 外部連携 | Google Workspace API（Sheets/Calendar/Drive/Gmail） |

## 6. 機能要件（詳細）

各機能に ID を付し、`MUST`（必須）/`SHOULD`（推奨）/`COULD`（任意・将来）で優先度を示す。

### F1. メンバー・組織管理
| ID | 要件 | 優先度 |
|---|---|---|
| F1-1 | メンバー名簿（氏名・ふりがな・所属委員会・役職・入会年度・連絡先・LINEユーザーID・有効/退会フラグ）をDBで一元管理する | MUST |
| F1-2 | 委員会・役職・班などの組織構造を保持し、配信対象の絞り込み（例：「総務委員会のみ」「五役のみ」）に使える | MUST |
| F1-3 | Google Sheetsの名簿と双方向に同期／一括インポート・エクスポートできる | SHOULD |
| F1-4 | 管理画面からメンバーのCRUD・一括編集ができる | MUST |
| F1-5 | LINE未連携メンバーを検出し、連携を促せる（招待リンク/QR発行） | SHOULD |

### F2. LINE連携・本人紐付け基盤
| ID | 要件 | 優先度 |
|---|---|---|
| F2-1 | LINE Webhookを受信し、メッセージ/ポストバック/友だち追加イベントを処理する | MUST |
| F2-2 | 友だち追加時に本人確認フローを実行し、LINEユーザーIDと名簿メンバーを紐付ける。**方式＝個別招待コード/QR（確定）**：名簿メンバーごとに一意の招待コード/QRを発行し、友だち追加後に入力させて照合・紐付けする。コードは有効期限・使い切り（再発行可）とする | MUST |
| F2-3 | リッチメニューで主要操作（出欠回答・予定確認・問い合わせ・資料提出）に導線を作る | SHOULD |
| F2-4 | クイックリプライ／ボタンテンプレートで誤入力の少ない回答UIを提供する | MUST |
| F2-5 | 1対1トークを基本とし、グループトークでの誤作動を防ぐ（メンション時のみ反応等） | SHOULD |
| F2-6 | 配信失敗（ブロック・退会・上限）を検知し記録、再送/代替手段にフォールバックする | MUST |

### F3. イベント・スケジュール管理
| ID | 要件 | 優先度 |
|---|---|---|
| F3-1 | 例会・委員会・理事会・各種イベントを登録（日時・場所・対象者・出欠締切・資料締切・概要）できる | MUST |
| F3-2 | Google Calendarと連携してイベントを取込／反映できる | SHOULD |
| F3-3 | イベントごとに対象者（全員/委員会別/役職別/任意指定）を設定できる | MUST |
| F3-4 | 繰り返しイベント（毎月例会等）のテンプレート化ができる | SHOULD |
| F3-5 | 締切・開催前の自動リマインド日程を設定できる（例：1週間前/前日/当日朝） | MUST |

### F4. 出欠確認・集計・催促
| ID | 要件 | 優先度 |
|---|---|---|
| F4-1 | 対象者へLINEで出欠確認を自動送信する。**ステータスは実運用に準拠＝出席 / Web出席 / 欠席（+ 遅刻早退、委任、未回答）**、欠席時は理由（体調不良/家庭の事情/仕事/冠婚葬祭/自由記述）を取得（`docs/drive-analysis.md` §9.1） | MUST |
| F4-2 | 回答をリアルタイムに集計し、現在の出欠状況・回答率を即時表示する | MUST |
| F4-3 | 未回答者に対し、設定スケジュールで段階的に自動催促する（締切までの逓増催促） | MUST |
| F4-4 | 回答変更（出席→欠席等）を受け付け、履歴を残す | MUST |
| F4-5 | 委任状・代理出席・定足数（理事会等）の判定を補助する | SHOULD |
| F4-6 | 集計結果をGoogle Sheets/CSVに出力、五役へサマリ自動通知する | MUST |
| F4-7 | 自然文（「来週の例会は欠席で」）からの出欠意図をGeminiで解釈し確認のうえ登録する | SHOULD |

### F5. 対外連絡（ブロック協議会等）の伝達
| ID | 要件 | 優先度 |
|---|---|---|
| F5-1 | ブロック協議会等からの連絡を取り込む。**主経路＝専用Gmailアドレスへの転送（確定）**：受信連絡を専用アドレスに転送するとGmail API/Pub-Sub経由で自動取込・本文/添付（PDF等）を解析する。管理画面アップロードは補助手段 | MUST |
| F5-2 | 連絡内容をGeminiで要約・整形し、メンバー向けにわかりやすい告知文を生成する | MUST |
| F5-3 | 連絡から「対象者・期限・必要なアクション（参加登録/提出/出向 等）」を抽出し、タスク化する | SHOULD |
| F5-4 | 適切な対象（全員/特定委員会/出向者）へ自動配信する | MUST |
| F5-5 | アクションが必要な連絡は、対象者の対応状況を追跡し未対応者を催促する | SHOULD |
| F5-6 | 原文（出典）への参照を残し、要約と原文を切り分けて提示する（誤要約対策） | MUST |

### F6. 会議資料の提出管理・内容レビュー
| ID | 要件 | 優先度 |
|---|---|---|
| F6-1 | 会議（理事会/五役会/委員会）ごとに、提出すべき資料・担当・締切を定義する | MUST |
| F6-2 | LINE/管理画面/Driveから資料を受領し、提出状況（提出済/未提出/差戻し）を管理する | MUST |
| F6-3 | 未提出担当者へ締切前後で自動催促する | MUST |
| F6-4 | **形式チェック**：必須項目（日時・場所・目的・予算・対象者・議案番号等）の記載漏れ、所定フォーマット逸脱を検出する。**標準テンプレート（上程議案・事業計画書等）が存在するため、これを正規フォーマットとして定義しチェックの基準にする（確定）** | MUST |
| F6-5 | **内容レビュー（LLM）**：Geminiで議案の要約・論点抽出、過去議案や上位方針との整合確認、曖昧・矛盾箇所や要確認点の指摘を行う。事業計画書の「09 検証/12 討議協議審議のポイント/13 ブラッシュアップ点」等を重点レビュー対象とする。表記規定（**英数記号は半角**：`docs/drive-analysis.md` §4）の逸脱も自動検出する | MUST |
| F6-6 | レビュー結果を担当者・専務へフィードバックし、修正→再提出のループを回す | MUST |
| F6-7 | 全提出資料を会議単位でまとめ、事前共有用パッケージ（目次付き）を生成する | SHOULD |
| F6-8 | LLMレビューは「助言」であり最終判断は人間が行う旨を明示する（責任分界） | MUST |

### F7. 例会アンケートの実施・集計・催促
| ID | 要件 | 優先度 |
|---|---|---|
| F7-1 | 例会等の後にアンケートを配信する。**現行はGoogleフォームで対内/対外の2系統運用（5段階Likert＋自由記述、`docs/drive-analysis.md` §9.2）**。対外（非会員）はLINE不可のためフォーム維持、対内はLINE/フォームを選択。エージェントは回答をForms APIで取得・集計する | MUST |
| F7-2 | 回答を集計し、スコア集計・自由記述のGeminiによる要約／分類／感情傾向を提示する | MUST |
| F7-3 | 未提出者を特定し、段階的に自動催促する | MUST |
| F7-4 | 集計レポート（定量＋定性サマリ）を生成し、五役・担当委員会へ自動通知する | MUST |
| F7-5 | 過去回との比較・経時トレンドを表示する | SHOULD |

### F8. 自然言語対話・問い合わせ応答
| ID | 要件 | 優先度 |
|---|---|---|
| F8-1 | メンバーからの自由文の問い合わせ（次回例会いつ？／会場どこ？／自分の出欠状況は？等）にGeminiで応答する | MUST |
| F8-2 | 回答はDB/Drive等の自LOM情報に基づき生成（RAG）し、根拠の薄い断定を避ける | MUST |
| F8-3 | エージェントが扱えない依頼は専務/担当へエスカレーション（取次・通知）する | MUST |
| F8-4 | 個人情報（他者の連絡先等）の不適切な開示を拒否するガードを持つ | MUST |
| F8-5 | 会話の文脈（直近のやり取り）を保持して自然に応答する | SHOULD |

### F9. 全自動配信エンジン ＋ ガードレール（重要）
全自動配信を選択したため、誤配信・過剰配信を防ぐ安全機構を必須とする。
| ID | 要件 | 優先度 |
|---|---|---|
| F9-1 | 配信は対象・本文・送信時刻を確定したジョブとしてキュー（Cloud Tasks）に積み、冪等に実行する（重複送信防止） | MUST |
| F9-2 | **静音時間（クワイエットアワー）**：深夜早朝（例 21:00〜8:00）は配信を保留し翌朝送信する | MUST |
| F9-3 | **レート制限／スロットリング**：単位時間あたり・1人あたりの配信上限を設け、催促の過剰連投を防ぐ | MUST |
| F9-4 | **サニティチェック**：対象人数が想定外に多い・本文が空/異常に長い・プレースホルダ未置換 等を検知したら自動配信を止め監督者に通知する | MUST |
| F9-5 | **キルスイッチ**：監督者が全配信を即時停止できる（グローバル/機能別） | MUST |
| F9-6 | **ドライラン／プレビュー**：配信前の対象・本文を監督者が随時確認できる（任意の事前承認モードへ切替も可能に） | SHOULD |
| F9-7 | 全配信の監査ログ（誰に・いつ・何を・結果）を保存し追跡可能にする | MUST |
| F9-8 | LLM生成文には軽量な内容ポリシーチェック（個人情報露出・不適切表現・事実逸脱）を通す | SHOULD |

### F10. 管理ダッシュボード
| ID | 要件 | 優先度 |
|---|---|---|
| F10-1 | イベント別の出欠状況・回答率・未回答者一覧を可視化する | MUST |
| F10-2 | 資料提出状況・レビュー結果を一覧表示する | MUST |
| F10-3 | 配信ログ・予定配信・失敗配信を確認/再送できる | MUST |
| F10-4 | ガードレール設定（静音時間・上限・催促スケジュール・キルスイッチ）を変更できる | MUST |
| F10-5 | メンバー名簿・組織・イベントを編集できる | MUST |
| F10-6 | IAP/Firebase Auth等で役員のみアクセス可能（ロール別権限） | MUST |

### F11. スケジューラ・リマインド基盤
| ID | 要件 | 優先度 |
|---|---|---|
| F11-1 | Cloud Schedulerで定期的に「締切判定→催促対象抽出→配信ジョブ生成」を実行する | MUST |
| F11-2 | 催促ポリシー（何日前/何回/間隔）をイベント種別ごとに設定できる。**理事会の初期値（最新2026アジェンダより）＝エントリー7日前24:00 / 資料提出5日前24:00 / 資料配信3日前24:00**を基準に逓増催促する。締切は年度ごとにアジェンダ「確認事項」から自動取得すると堅牢（詳細は `docs/drive-analysis.md` §2・§11） | MUST |
| F11-3 | タイムゾーンは Asia/Tokyo 固定で扱う | MUST |

### F12. レポーティング・通知
| ID | 要件 | 優先度 |
|---|---|---|
| F12-1 | 五役向けの定例サマリ（今週の予定・出欠状況・未提出資料・要対応連絡）を自動生成・配信する | SHOULD |
| F12-2 | 各レポートはGoogle Sheets/Docs/PDFへ出力できる | COULD |

## 7. 非機能要件

| カテゴリ | 要件 |
|---|---|
| セキュリティ | 通信TLS、Secret ManagerでAPIキー管理、最小権限IAM、管理画面はIAP保護、LINE Webhookは署名検証必須 |
| 個人情報保護 | 取得目的の明示と同意取得、利用範囲限定、退会者データの削除/匿名化、アクセスログ保持、LLMへ渡す個人情報の最小化 |
| 可用性 | サーバレス（Cloud Run）でスケール、想定規模では単一リージョンで可。配信は再試行付きキューで担保 |
| 性能 | LINE応答は数秒以内（即時ackし重い処理は非同期化）。配信は分単位の遅延許容 |
| 監査性 | すべての自動配信・LLM判断・設定変更を監査ログに記録 |
| 保守性 | 設定（催促ポリシー・テンプレ・静音時間）はコード変更なしで管理画面から変更可能 |
| コスト | サーバレス従量課金前提。数十名規模で月額を低く抑える（Cloud Run/Firestore/Geminiとも小規模利用） |
| 国際化 | 日本語前提。日時はJST |
| LLM安全性 | プロンプトインジェクション対策、出典提示、断定回避、人間最終判断の明示 |

## 8. データモデル概要（主要エンティティ）

- **Member**: id, lom_id, 氏名, ふりがな, 役職, 委員会, 入会年度, 連絡先, line_user_id, status(active/退会)
- **Organization/Committee**: id, 名称, 種別, 親組織, メンバー一覧
- **Event**: id, 種別(例会/理事会/委員会/イベント), 日時, 場所, 対象範囲, 出欠締切, 資料締切, 状態
- **Attendance**: id, event_id, member_id, status(出席/欠席/遅刻/未定/委任), 理由, 回答日時, 履歴
- **ExternalNotice**: id, 出典, 受領日, 原文, 要約, 対象, 必要アクション, 期限, 状態
- **Document/Submission**: id, event_id, 担当, 種別, 締切, 状態(未/済/差戻), storage_uri, レビュー結果
- **Survey / SurveyResponse**: アンケート定義・回答・集計
- **DeliveryJob / DeliveryLog**: 配信予定・結果（監査ログ）
- **Policy/Setting**: 催促ポリシー, 静音時間, レート上限, テンプレート, キルスイッチ状態

## 9. LLM（Gemini）活用方針

| 用途 | 概要 |
|---|---|
| 対外連絡の要約・告知文生成 | 原文→平易な告知文。原文参照を併記し誤要約を抑制 |
| 会議資料の内容レビュー | 要約・論点抽出・整合確認・不備指摘。最終判断は人間 |
| アンケート自由記述の分析 | 要約・分類・傾向把握 |
| 自然言語の出欠/問い合わせ解釈 | 意図抽出→確認のうえDB登録（RAGで自LOM情報に接地） |
| 配信文のポリシーチェック | 個人情報露出・不適切表現の軽量検査 |

方針：**自LOMのデータに接地（RAG）**し、断定を避け、重要判断は人間に委ねる。LLMへ渡す個人情報は最小化する。

## 10. リスクと対策（特に全自動配信）

| リスク | 対策 |
|---|---|
| 誤った対象/本文の自動配信 | F9 サニティチェック・冪等ジョブ・ドライラン・キルスイッチ・監査ログ |

| プロンプトインジェクション（外部文書経由） | 入力の信頼境界分離、システム指示の固定、出力検査 |

## 12. 確定事項・未決事項

### 12.1 確定事項（v0.1で決定済み）
- **対象範囲**：単一LOM（〜数十名）。複数LOMは将来拡張。
- **正データ**：GCP側DBを正とし、Google Workspaceと連携するハイブリッド。
- **配信自律性**：全自動配信（承認不要）＋ガードレール必須。
- **資料チェック**：提出管理＋Geminiによる内容レビューまで。
- **本人確認**：個別招待コード/QRによる紐付け（F2-2）。
- **対外連絡の取込**：専用Gmailアドレスへの転送が主経路（F5-1）。
- **資料テンプレート**：標準テンプレートあり。これを形式チェックの基準とする（F6-4）。
- **出欠/アンケート方式＝ハイブリッド**：出欠はLINEネイティブ（クイックリプライ＋自動催促）、アンケートは対外（非会員）をGoogleフォーム維持・対内は選択。エージェントはForms APIで回答取得・集計・未回答者をLINE催促（`docs/drive-analysis.md` §9・§12）。
- **現行運用の前提**：このLOM（JCI猪苗代）は既にGoogleフォームで出欠/アンケートを月次運用中。会員名簿はxlsx（カード型）。MVPはこの資産と接続/置換する形で設計。
